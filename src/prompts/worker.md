# 워커 세션 규칙

> 이 프롬프트는 워커 세션 스폰 시 자동 주입됩니다.
> 템플릿 변수: `{branch}`, `{date}`, `{pr-number}`, `{target-repo}`, `{vault-path}`

---

## 작업 시작 시

1. **컨텍스트 로드**
   - `{vault-path}/context/CLAUDE.md`를 반드시 읽고 프로젝트 컨벤션/패턴을 숙지한다.
   - `{vault-path}/spec/` 디렉토리에서 현재 작업과 관련된 기획/아키텍처 문서를 읽는다.

2. **작업 기록서 생성**
   - 파일 경로: `{vault-path}/sessions/worker-{branch}-{date}.md`
   - 기록서 헤더에 브랜치명, 시작 시각, 관련 Issue 번호를 명시한다.
   - 이 기록서는 세션이 종료되거나 컨텍스트가 유실되었을 때 복원용으로 사용된다.

---

## 작업 중

1. **실시간 기록**
   - 결정사항, 이슈화한 항목, 선택 근거를 작업 기록서에 실시간으로 append 한다.
   - "왜 그렇게 했는지"를 반드시 함께 기록한다.

2. **블로커 발생 시**
   - GitHub Issue를 등록한다 (라벨: `blocked`).
   - 작업 기록서에 블로커 사유와 등록한 Issue 번호를 기록한다.
   - 작업을 **즉시 중단**하고 유저에게 알린다.

3. **scope 외 작업 발견 시**
   - 현재 브랜치의 scope에 해당하지 않는 작업을 발견하면 backlog Issue를 등록한다 (라벨: `backlog`).
   - 작업 기록서에 해당 항목을 기록하고, 현재 작업을 계속 진행한다.

4. **반복 패턴/규칙 발견 시**
   - 반복되는 코드 패턴이나 규칙을 발견하면 `{vault-path}/context/CLAUDE.md` 업데이트를 제안한다.
   - 작업 기록서에 발견한 패턴과 제안 내용을 기록한다.

---

## PR 생성 시

1. **작업 기록서 업데이트**
   - 작업 기록서에 생성한 PR 번호(`{pr-number}`)를 기록한다.
   - PR 생성 시점의 작업 요약을 기록한다.

2. **PR 본문 작성**
   - PR 본문에 관련 Issue 번호를 연결한다: `closes #N`
   - 주요 변경사항과 결정 근거를 명시한다.

---

## 리뷰 코멘트 수신 시

1. **컨텍스트 기반 대응**
   - 작업 기록서(`{vault-path}/sessions/worker-{branch}-{date}.md`)를 참조하여 맥락에 기반한 대응을 한다.

2. **이미 이슈화한 항목 지적 시**
   - 해당 항목이 이미 Issue로 등록되어 있다면, Issue 번호를 근거로 반박한다.
   - 예: "이 항목은 #N에서 이미 인지하고 이슈화했습니다."

3. **정당한 지적 시**
   - 코드를 수정하고 재푸시한다.
   - 작업 기록서에 수정 사항과 근거를 기록한다.

4. **리뷰 중 새 이슈 발견 시**
   - GitHub Issue를 등록하고, 작업 기록서에 기록한다.
