# 디버깅 가이드 — Speedy vs Focus 모드

## 개요

버그/이슈 분석에는 두 가지 모드가 있다.
기본은 **Speedy**. 근본 원인을 깊이 파야 할 때만 **Focus**로 전환한다.

---

## Speedy (기본)

> 빠르게 문제를 정의하고 해결책을 제시한다.

worker.md / reviewer.md에 정의된 서식 그대로 사용:

```
## 1. 버그 명세

**발생 위치**: 파일명:라인번호, 함수명

**이 코드는 어디에 있고, 무엇을 하는 곳인가**:
- 시스템에서의 역할 (어떤 기능의 어떤 부분)
- 왜 존재하는지 (상위/하위 요구사항)
- 호출 체인

**왜 문제인가**:
유저/시스템에 미치는 실제 영향

**버그 원인**:
입력 → 처리 → 잘못된 결과

## 2. 해결책

**방안 A / B**: 코드 예시 + 장단점 + 추가 이슈
```

**언제 쓰나**: 원인이 명확하고, 수정 범위가 좁을 때.

---

## Focus

> 해결책을 제시하지 않는다. 문제의 근본 원인만 깊이 파고든다.

Speedy와 다른 점:
- **해결책 섹션 없음** — 문제 이해가 목적
- **추가 분석 섹션 있음** — 연쇄 영향, 설계적 원인, 유사 패턴

```
## 1. 버그 명세

(Speedy와 동일)

## 2. 근본 원인 분석

**표면적 원인**:
직접적으로 버그를 발생시키는 코드/로직

**설계적 원인**:
왜 이런 코드가 작성될 수밖에 없었는지.
인터페이스 설계, 타입 시스템, 아키텍처 수준의 원인.

**연쇄 영향**:
이 문제가 다른 곳에도 같은 패턴으로 존재하는지.
| 위치 | 동일 패턴 여부 | 영향 |
|------|---------------|------|
| ... | ... | ... |

**재발 조건**:
어떤 상황에서 이 버그가 다시 만들어질 수 있는지.
향후 코드 변경 시 주의점.

## 3. 미해결 질문

- 이 문제를 해결하려면 먼저 답해야 하는 질문들
- 유저에게 확인받아야 하는 설계 결정
```

**언제 쓰나**:
- 원인이 불명확할 때
- 같은 패턴의 버그가 반복될 때
- 수정이 다른 곳에 연쇄 영향을 줄 때
- 인터페이스/아키텍처 수준의 변경이 필요할 수 있을 때

---

## 판단 기준

| 상황 | 모드 |
|------|------|
| 오타, 누락, 단순 로직 에러 | Speedy |
| 한 파일 내에서 해결 가능 | Speedy |
| 여러 어댑터에 걸친 시맨틱 불일치 | Focus |
| "왜 이렇게 설계했지?" 의문이 들 때 | Focus |
| 수정했는데 또 비슷한 버그가 나올 것 같을 때 | Focus |
| 인터페이스 변경이 필요할 수 있을 때 | Focus |

---

## 예시: 같은 버그를 두 모드로

### Speedy 버전

> `list()`의 labels 필터가 Jira에서 AND, Local에서 OR로 동작한다.
> `getAllIssueStatuses()`가 OR을 기대하므로 Jira에서 빈 배열 반환.
> 방안 A: OR로 통일, 방안 B: 문서화만

### Focus 버전

> `list()`의 labels 필터가 어댑터마다 다르게 동작한다.
>
> **설계적 원인**: `ListIssuesOptions.labels`의 시맨틱이 인터페이스에 정의되지 않았다.
> 각 어댑터가 자의적으로 구현했고, 호출자(`getAllIssueStatuses`)는 OR을 가정한다.
>
> **연쇄 영향**: GitHub도 AND로 동작하여 동일 버그 존재.
> Jira는 마이그레이션으로 `statuses` 분기를 타서 우회됐지만 근본 해결 아님.
>
> **미해결 질문**: labels의 표준 시맨틱을 OR로 확정할 것인가?
> GitHub `gh` CLI가 OR을 지원하지 않는데 어떻게 구현할 것인가?
